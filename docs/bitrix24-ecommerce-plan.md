# План и требования: полноценный магазин + интеграция с Битрикс24

**Статус:** проработка (без реализации в коде); **согласовано с ТЗ 2.0** (каталог и остатки из Б24, заказы на сайте → Б24).  
**Цель:** зафиксировать направление работ, границы ответственности систем и критерии, по которым потом можно оценивать смету и архитектуру. Сводка ТЗ: [tz-2-0-alignment.md](./tz-2-0-alignment.md).

---

## 1. Исходные условия (как понимаем задачу)

| Источник | Роль |
|----------|------|
| **Битрикс24**, редакция с **интернет-магазином и CRM** | Учёт товаров (**цены, остатки**), связка с **Wildberries и Ozon** (маркетплейсы как каналы продаж/учёта — детали уточнять у внедренца Б24). |
| **Текущий проект** (Vite/React + Django) | Должен стать **полноценным интернет-магазином** на собственном фронте/бэке: каталог, корзина, оформление, оплата, доставка, ЛК покупателя, админка владельца. |
| **Интеграция** | Заказы, оформленные **на сайте**, должны **попадать в Битрикс24** (в понятную сущность: сделка / заказ / лид — см. раздел 4). |

**Важно:** «1С-Битрикс: Управление сайтом» и **Битрикс24** — разные продукты. У вас указан **Битрикс24**; API, сущности и лимиты будут от **облачного портала Б24** и включённых приложений. План ниже опирается на **Битрикс24 REST API** и типовые сценарии CRM; конкретные методы и объекты магазина в Б24 нужно **валидировать** с документацией и тарифом клиента.

## 2. Ключевые архитектурные вопросы (нужно решить до проектирования)

### 2.1. Где «истина» по каталогу и остаткам?

| Вариант | Плюсы | Минусы |
|---------|--------|--------|
| **A. Единственный источник — Б24** | Один склад учёта; WB/Ozon уже «смотрят» в Б24. | Сайт постоянно зависит от API Б24 (лимиты, задержки); нужен кэш и стратегия при недоступности Б24. |
| **B. Единственный источник — наш backend** | Быстрый каталог, полный контроль UX. | Дублирование данных с Б24; риск расхождений с МП; нужна **двусторонняя или однонаправленная синхронизация**. |
| **C. Гибрид** | Контент/SEO на сайте, **цены и остатки** из Б24 (или наоборот: витрина у нас, синк в Б24 раз в N минут). | Сложнее поддерживать; нужны чёткие правила конфликтов. |

**Рекомендация для планирования:** заложить **вариант A или C** по умолчанию, если клиент уже ведёт товар в Б24 ради МП — иначе расхождения «сайт vs Б24 vs WB» неизбежны.

### 2.2. Что именно «уходит» в Битрикс24 при заказе?

Минимально различимые цели:

- **CRM:** контакт/компания, **сделка** или **лид** с суммой, составом, статусом оплаты/доставки.
- **Если в Б24 реально используется модуль заказов магазина** (в рамках их редакции): создание **заказа** с позициями, привязка к контрагенту.

**Требование к уточнению:** одна встреча/чек-лист с администратором Б24: *куда кладём заказ с сайта, какие поля обязательны, нужны ли товары из каталога Б24 по `XML_ID` / артикулу.*

### 2.3. Оплата (эквайринг)

- **На сайте** обычно: платёжный провайдер (ЮKassa, Тинькофф, Сбер и т.д.) → **webhook** на наш backend → смена статуса заказа → уведомление/синк в Б24.
- **Вопрос:** нужна ли **оплата внутри Б24** или только учёт факта оплаты, пришедшего с сайта.

**Требование:** зафиксировать **одного** основного провайдера эквайринга и схему (одностадийная / двухстадийная, возвраты).

### 2.4. Доставка (СДЭК)

Типовой набор:

- Расчёт тарифов и сроков (город/ПВЗ).
- Создание **заказа на доставку** в API СДЭК после оплаты или после подтверждения менеджером (политика бизнеса).
- Трекинг; при необходимости — проброс статуса в Б24.

**Требование:** определить момент создания отправления в СДЭК (сразу после оплаты / после сборки / вручную из админки).

## 3. Функциональные требования (черновик для ТЗ)

### 3.1. Витрина и магазин (сайт)

- Каталог с **актуальными** ценой и наличием (источник — по решению п. 2.1).
- Карточка товара, корзина, оформление заказа (контакты, доставка, оплата).
- Юридические страницы: оферта, политика ПДн, доставка/оплата.
- Адаптив, производительность (как в текущем ТЗ).

### 3.2. Личный кабинет покупателя

- Регистрация / вход (email+пароль, опционально телефон + SMS — отдельное ТЗ и бюджет).
- История заказов, статусы (оплачен, собран, передан в доставку, доставлен).
- Повтор заказа / избранное (по приоритету).
- **Синхронизация идентичности с Б24:** либо только наш `user_id` + поля в сделке, либо поиск/создание контакта в Б24 по email/телефону (правила дедупликации).

### 3.3. Админка владельца

Сейчас — Django admin; для «полноценного магазина» обычно нужно:

- Заказы: фильтры, смена статуса, комментарии, печать/экспорт.
- Товары (если каталог на сайте): либо **только просмотр + ручные правки контента**, либо полный CRUD (если не Б24-only).
- Настройки доставки/оплаты (без правки кода).
- Журнал ошибок интеграций (Б24, СДЭК, платежи).

**Требование:** определить **MVP админки** (только заказы + логи интеграций) vs полная кастомная панель.

### 3.4. Интеграция с Битрикс24

- **Исходящие события с сайта:** создание/обновление сущности в Б24 при оформлении заказа, при смене статуса оплаты, (опционально) при смене статуса доставки.
- **Аутентификация:** OAuth приложения в Б24, хранение токенов на backend, обновление refresh_token.
- **Идемпотентность:** повтор webhook платежки не должен создавать дубликаты сделок в Б24.
- **Ошибки:** очередь повторов, алерты, ручное «досоздать в Б24» из админки.

### 3.5. Интеграция с маркетплейсами (граница ответственности)

- Если **WB/Ozon полностью ведутся из Б24**, сайт **не обязан** напрямую интегрироваться с МП — только с Б24.
- Если позже понадобится **единая витрина цен** — это отдельный эпик после стабилизации Б24.

---

## 4. Нефункциональные требования (черновик)

- Доступность API Б24: при таймауте — деградация (очередь, не потерять заказ локально).
- Безопасность: секреты только на сервере; PCI DSS — данные карт не хранить (только токены провайдера).
- Аудит: кто изменил статус заказа, когда ушло в Б24.
- Соответствие 152-ФЗ: ПДн в политике, согласия, возможность выгрузки/удаления по запросу (уровень — согласовать с юристом).

## 5. Поэтапный план (логика внедрения)

| Фаза | Содержание | Результат |
|------|------------|-----------|
| **0. Discovery** | Уточнить редакцию Б24, где хранятся товары/остатки, как заведены WB/Ozon, пример «идеального» заказа в Б24; выбрать эквайринг; сценарий СДЭК. | Документ «Как заказ с сайта выглядит в Б24» + схема полей. |
| **1. MVP заказа** | Оформление на сайте → сохранение в **нашем** backend → один REST-вызов в Б24 (сделка/заказ) → админка списка заказов и статуса синка. | Первые реальные заказы в Б24 без ЛК и без оплаты *или* с простой оплатой. |
| **2. Оплата** | Подключение эквайринга, webhooks, статусы, связка с заказом и повторная отправка в Б24 при оплате. | Закрытый цикл «оплатил → видно в Б24». |
| **3. СДЭК** | Калькулятор, выбор ПВЗ/курьер, создание накладной, статусы. | Доставка end-to-end на сайте. |
| **4. ЛК покупателя** | Регистрация, история заказов, привязка к контакту Б24. | Самообслуживание. |
| **5. Каталог из Б24** (если выбрано) | Синхронизация номенклатуры/остатков/цен, кэш, расписание. | Единый склад учёта с МП. |
| **6. Админка** | Расширение под операционные задачи, отчёты, роль менеджера. | Удобство владельца. |

Фазы 1–3 можно частично переставлять по приоритету клиента (например, сначала СДЭК без онлайн-оплаты).

## 6. Риски и открытые вопросы

1. **Лимиты и стоимость** REST Битрикс24 на тарифе клиента; нужен ли отдельный **серверный** IP для webhook’ов.
2. **Соответствие номенклатуры:** SKU на сайте = товар в Б24 (артикул, внешний код).
3. **Частичные отгрузки и возвраты** — как отражать в Б24 при споре с МП.
4. Нужна ли **двусторонняя** синхронизация статусов (менеджер закрыл сделку в Б24 → обновился статус на сайте).

---

## 7. Связь с текущим репозиторием

- Сейчас: Django API, заявки (корзина/калькулятор) как **лиды**, без оплаты и без полноценного заказа как в e-commerce.
- Переход к ТЗ из этого документа потребует: модель **заказа** (статусы, позиции, платежи, доставка), **пользователи**, **платежный модуль**, **модуль СДЭК**, **сервисный слой Б24** — объём сопоставим с новым крупным эпиком.

## 8. Рекомендуемые следующие шаги (не разработка)

1. Заполнить **чек-лист discovery** с администратором Б24 (скрин воронки, поля сделки, тестовый заказ вручную).
2. Получить от клиента **доступ к песочнице** Б24 или тестовому порталу + заявку на приложение OAuth.
3. Выбрать **платёжного оператора** и получить тестовые ключи.
4. Утвердить **MVP** по разделу 5 (что входит в первый контракт/спринт).
