# Два больших этапа: магазин + CRM, затем импорт номенклатуры

Цель **этапа 1** — опубликовать **интернет-магазин** с **личными кабинетами**, **эквайрингом**, **доставкой** (СДЭК или иной выбранный провайдер по ТЗ) и **актуальным каталогом**; все **заказы, оплаты и данные покупки**, необходимые для CRM, **уходят в Битрикс24**.

Цель **этапа 2** (после запуска в проде) — стабилизировать **источник правды по товарам**: синхронизация **из Битрикс24** и/или **отдельные каналы Wildberries и Ozon**, без дублирования конфликтующих «истин».

Связанные документы: [tz-2-0-alignment.md](./tz-2-0-alignment.md), [bitrix24-ecommerce-plan.md](./bitrix24-ecommerce-plan.md), [excel-catalog-import.md](./excel-catalog-import.md).

## Этап 1. Интернет-магазин и связка с Битрикс24

Рекомендуемый порядок: сначала **контуры интеграции и модели данных**, затем **витрина и ЛК**, затем **доставка и оплата**, затем **сквозной заказ**.

### Блок A. Входные решения и проектирование (1–2 недели, параллельно с разработкой)

| Шаг | Действие | Результат |
|-----|----------|-----------|
| A.1 | Зафиксировать **тариф и возможности** портала Б24: есть ли **интернет-магазин / заказы**, **склад**, **контакты**, лимиты REST | Письменный чек-лист «что доступно в API» |
| A.2 | Решить **куда падает заказ**: сделка, смарт-процесс, заказ магазина Б24; обязательные поля, статусы | Схема сущности + маппинг статусов сайт ↔ Б24 |
| A.3 | Решить **идентификатор товара**: `XML_ID`, артикул, SKU МП; связь позиции корзины с позицией в Б24 | Таблица маппинга |
| A.4 | Выбрать **эквайринг** (ЮKassa, банк, Ozon Acquiring и т.д.): одностадийно/двухстадийно, СБП, чеки | Договор + тестовые ключи |
| A.5 | СДЭК: тестовый кабинет, договор по **API v2**, откуда забирать отправитель (Иваново) | Ключи в `.env`, не в Git |
| A.6 | **Postgres + Redis** на проде; домен, HTTPS, почта для уведомлений | Инфраструктурный чек-лист |

### Блок B. Backend: фундамент и зеркало данных

| Шаг | Действие | Результат |
|-----|----------|-----------|
| B.1 | Миграции: пользователь покупателя, профиль, адреса; **локальное зеркало заказа** (статусы, трек, суммы, id Б24) | Модели + индексы |
| B.2 | Сервис **синка каталога** из Б24 → локальный кэш (товар, цена, остаток, картинки по URL из Б24 при необходимости); **оверлей** из ТЗ: `visible`, `sort_order`, SEO | Celery-задача + webhook «товар изменён» (если доступен) |
| B.3 | Клиент REST Б24: получение товаров/остатков, **создание заказа/сделки**, обновление статуса; ретраи и **очередь несинхронизированных** заказов | Модуль `integrations/bitrix24/` (или аналог) |
| B.4 | **Входящие вебхуки** Б24: смена статуса заказа/сделки → обновление локального зеркала и отображение в ЛК | HTTPS endpoint + подпись/секрет |
| B.5 | JWT (или сессии) для покупателя; связь **email/телефон** с контактом в Б24 (правило дедупликации) | Auth API + документирование |

### Блок C. Витрина и корзина (frontend + API)

| Шаг | Действие | Результат |
|-----|----------|-----------|
| C.1 | Каталог и карточка читают **кэш**; фильтры/сортировка согласованы с полями из Б24 | UI без «расхождений цены» |
| C.2 | Корзина (Zustand/сервер): состав с **id товара Б24** / внутренним ключом маппинга | Сохранение до оформления |
| C.3 | Многошаговый checkout: контакты → доставка → оплата → подтверждение | Макеты + валидация |

### Блок D. Доставка (СДЭК)

| Шаг | Действие | Результат |
|-----|----------|-----------|
| D.1 | Расчёт тарифов и сроков (город/ПВЗ) API | Endpoint для фронта |
| D.2 | Список ПВЗ + отображение на **Яндекс.Картах** (API-ключ карты в `.env`) | Выбор ПВЗ в checkout |
| D.3 | После оплаты (или по политике — после подтверждения менеджера): **создание отправления** в СДЭК, сохранение трека | Связка с заказом в БД и полями в Б24 |

### Блок E. Оплата (эквайринг)

| Шаг | Действие | Результат |
|-----|----------|-----------|
| E.1 | Создание платежа на сумму заказа, **return URL**, метаданные (номер заказа) | Интеграция SDK/API провайдера |
| E.2 | Webhook оплаты → смена статуса локально → **фиксация оплаты в Б24** (поле/стадия) | Идемпотентная обработка webhook |
| E.3 | Сценарий **отмены/возврата** и несостоявшейся оплаты | Документировано |

### Блок F. Сквозной сценарий «Заказ улетел в Б24»

| Шаг | Действие | Результат |
|-----|----------|-----------|
| F.1 | **Атомарность**: черновик заказа локально → при успехе оплаты (или по выбранной политике) — **создание в Б24** с составом, ФИО, доставкой, способом оплаты | Логи и алерт при ошибке |
| F.2 | При падении Б24 — заказ с флагом `pending_bitrix`, Celery **до 5 попыток** (как в ТЗ) | Не терять деньги клиента без учёта в CRM |
| F.3 | ЛК: история из **локального зеркала** + подтягивание статуса из Б24/webhook | Единый UX |

### Блок G. Админка и эксплуатация

| Шаг | Действие | Результат |
|-----|----------|-----------|
| G.1 | Просмотр заказов, фильтры, ручная смена статуса с **отправкой в Б24** (если предусмотрено API) | Минимально Django admin; целевое — React Admin по ТЗ |
| G.2 | Настройки интеграций (без секретов в БД в открытом виде — через env или шифрование полей) | Чек-лист секретов |
| G.3 | Мониторинг (Sentry), алерты на очередь несинкнутых заказов | Runbook |

### Блок H. Приёмка этапа 1

- Тестовый заказ: корзина → СДЭК → оплата тест → **запись в Б24** с верным составом и суммой.
- Смена статуса в Б24 → отображение в ЛК.
- Нагрузочно стыдливо: кэш каталога, rate limit API.

**Ориентир по срокам:** при одном middle и готовых доступах — порядка **8–14 недель** (согласуется с ТЗ 2.0), сильно зависит от сложности Б24 и выбранного эквайринга.

## Этап 2. После публикации: импорт / синхронизация товаров

Задача — **одна «истина»** по номенклатуре и остаткам, без ручного копирования.

### Вариант 2A. Единый источник — Битрикс24

| Шаг | Действие |
|-----|----------|
| 2A.1 | В Б24 завести/нормализовать каталог так, как он должен продаваться на сайте (группы, свойства, артикулы). |
| 2A.2 | Настроить **регулярный синк** (Celery + webhook): сайт только отображает + overlay (видимость, порядок, SEO). |
| 2A.3 | WB/Ozon не импортировать на сайт напрямую: они уже «питаются» от учёта в Б24 (если так настроено у внедренца). |

**Когда выбирать:** если склад и МП уже завязаны на Б24.

### Вариант 2B. Прямые каналы WB / Ozon на сайт (обход или дубль)

| Шаг | Действие |
|-----|----------|
| 2B.1 | Получить **API/отчёты** WB и Ozon (товары, остатки, цены — что доступно по договору). |
| 2B.2 | Построить **нормализатор** (единая модель SKU, атрибуты, штрихкод/артикул) и слияние с каталогом сайта **или** выгрузку в Б24 как промежуточный хаб. |
| 2B.3 | Разрулить **конфликты**: две цены, разный остаток → правило приоритета (например, остаток минимум из каналов, цена — с сайта из Б24). |

**Когда выбирать:** если Б24 не содержит полного каталога, а МП — основной учёт.

### Вариант 2C. Гибрид (часто в реальности)

- **Цены и остатки для витрины** — из Б24 после того, как туда попадают данные из МП (интеграция «МП → Б24» делает не сайт, а Б24/обмен).
- Сайт **никогда** не пишет остатки в WB напрямую без отдельного ТЗ.

Этап 2 обычно **1–4 недели** после стабилизации этапа 1, в зависимости от API и качества данных.

## Какой информации не хватает (нужно от вас / от внедренца)

Ниже — без этого нельзя финализировать смету и некоторые шаги A.x / F.x.

1. **Битрикс24:** точная редакция/приложения; куда именно создаём сущность при заказе (сделка vs заказ магазина); список **обязательных полей**; есть ли **исходящий вебхук** на изменение товара; лимиты REST на портале.
2. **Идентификация товара:** совпадает ли **артикул на сайте** с полем в Б24 и с SKU на WB/Ozon; есть ли **таблица соответствий** (как в Excel заказчика — большое число в колонке 5).
3. **Эквайринг:** выбранный провайдер и **песочница**; нужна ли **двухстадийная** оплата обязательно; кто **делает чеки** (агрегатор).
4. **СДЭК:** договорной тип клиента; **когда** создавать накладную (сразу после оплаты / после сборки); нужен ли **наложенный платёж** в первой версии.
5. **Политика заказа при ошибке Б24:** можно ли показать покупателю «успех», если деньги списались, но Б24 недоступен (юридически и по процессу — ручной дозапуск).
6. **WB / Ozon на этапе 2:** есть ли уже **личные кабинеты API**; нужен ли импорт **только для витрины** или ещё **обратная** выгрузка заказов (обычно нет).
7. **React Admin vs Django admin:** жёсткое требование ТЗ или допускается поэтапно (сначала Django admin для заказов и настроек).

Когда появятся ответы — можно сузить блоки B.3, F.1 и весь этап 2 до одного выбранного варианта (2A/2B/2C).
