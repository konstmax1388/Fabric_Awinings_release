# Интеграция заказов с Битрикс24 (Astrum «Заявки с сайта»)

После оформления заказа на сайте (`POST /api/leads/cart/`) Django отправляет данные во внешнее приложение **«Заявки с сайта»** ([документация провайдера](https://app-5.astrum.agency/documentation)), которое создаёт контакт и сделку/лид в **Битрикс24**.

## Что сделано в коде

- Сервис: `backend/api/services/astrum_crm.py` — сбор JSON по спецификации API и `POST` с заголовком `X-API-Key`.
- Вызов после создания `CartOrder`: `backend/api/views.py` (`CartOrderCreateView`), сразу после уведомления по email. Ошибки CRM **не отменяют** ответ **201** клиенту (заказ уже сохранён); статус синхронизации смотрите в админке.
- Поля заказа: `bitrix_sync_status`, `bitrix_entity_id` (id ответа посредника, часто внутренний id заявки в Astrum), `bitrix_sync_error`, `bitrix_sync_attempts`. В админке: в списке заказов колонка **«CRM (Битрикс24)»** (цветной статус, краткий текст ошибки); в карточке заказа сверху блок **«CRM: Битрикс24 (Astrum)»** со сводкой и полным текстом ошибки. В боковом меню — пункт **«Заказы: ошибка отправки в CRM»**.
- Повтор при сбое:  
  `python manage.py retry_astrum_crm_orders [--limit 50]`

## Настройка (рекомендуется: админка)

1. В Битрикс24 установите приложение **«Заявки с сайта»** из маркетплейса и получите **API-ключ** в панели приложения ([документация](https://app-5.astrum.agency/documentation)).
2. Узнайте **ID ответственного** сотрудника в Б24 для параметра `assigned_default` (целое число).
3. В Django Admin: **Настройки сайта** → раздел **«Битрикс24: заявки с сайта (Astrum)»**:
   - включите **«Включить отправку заказов в Битрикс24 (Astrum)»**;
   - вставьте **API-ключ** (поле скрыто как пароль);
   - укажите **ID ответственного**;
   - при необходимости измените URL endpoint, поведение контактов/сделки, префикс названия сделки, таймаут.

Пока галочка «Включить…» **выключена**, используются только переменные окружения (см. ниже), если они заданы. Если галочка **включена**, но ключ или ответственный не заполнены, сохранение блока в админке завершится с ошибкой валидации; интеграция **не** подставляет env (чтобы не смешивать источники).

### Переменные окружения (альтернатива или Docker без правки БД)

| Переменная | Описание |
|------------|----------|
| `ASTRUM_CRM_API_KEY` | Ключ из панели Astrum |
| `ASTRUM_CRM_ASSIGNED_DEFAULT` | ID ответственного (число) |
| `ASTRUM_CRM_API_URL` | По умолчанию `https://app-5.astrum.agency/api/order` |
| `ASTRUM_CRM_CONTACT_BEHAVIOR` | `SELECT_EXISTING` или `CREATE_ANYWAY` |
| `ASTRUM_CRM_ENTITY_BEHAVIOR` | По умолчанию **`CREATE_ANYWAY`** — новая сделка на каждый заказ. `ADD_TO_EXISTING` добавляет текст к уже открытой сделке того же клиента (как «Дополнительный запрос»). Также `IGNORE_EXISTING`. |
| `ASTRUM_CRM_DEAL_TITLE_PREFIX` | Префикс заголовка сделки |
| `ASTRUM_CRM_TIMEOUT` | Таймаут HTTP, секунды (5–120, по умолчанию 15) |

Если ни админка (включённая и заполненная), ни env не дают пару «ключ + ответственный», запросы в Astrum **не выполняются**.

**Docker Compose:** в `.env` можно задать переменные; в `docker-compose.yml` для сервиса `api` они пробрасываются (см. блок `environment`).

## Карта полей (заказ → API)

- **contact:** `customer_name`, `customer_phone`, `customer_email`.
- **deal.title:** `{ASTRUM_CRM_DEAL_TITLE_PREFIX} {order_ref}`.
- **deal.comments:** номер заказа, сумма, полный текст `manager_letter` (как в письме менеджеру).
- **deal.products:** из `lines[]` — `product_name` ← `title`, `price` ← `priceFrom`, `quantity` ← `qty`.
- **`product_id` (привязка к каталогу Б24):** передаётся **только** если в админке у товара/варианта заполнено поле **«ID товара в каталоге Битрикс24»** / **«ID в каталоге Б24 (вариант)»**. Логика: `backend/api/services/bitrix_product_map.py` находит товар по `productId` или `slug`, вариант по `variantId` (с фронта уходит вместе с заказом); приоритет у ID у **варианта**, иначе у **товара**. Раньше в Astrum ошибочно уходил внутренний id сайта — так больше не делается.

## Сопоставление товаров сайта и Битрикс24

**Автоматический поиск по названию в Б24 из этого репозитория не делается** — для этого нужны отдельные запросы к REST API портала (OAuth, права, лимиты) и политика совпадений (риск перепутать похожие названия). Практичный путь:

1. В CRM откройте нужную позицию **каталога товаров** и возьмите **числовой ID** сущности (часто виден в URL карточки или в экспорте — уточните у администратора Б24 под ваш тип каталога).
2. В Django Admin → **Товары** → карточка товара:
   - блок **«Битрикс24: сопоставление с каталогом CRM»** — общий ID для товара без отдельных вариантов или запасной;
   - в таблице **вариантов** — колонка **«ID в каталоге Б24 (вариант)»** для каждого SKU.
3. После сохранения при заказе в теле запроса к Astrum в позиции появится `product_id` с этим ID — посредник и Б24 смогут связать строку со **своим** товаром вместо «голого» текста (как описано в [документации Astrum](https://app-5.astrum.agency/documentation) для поля `products[].product_id`).

### Массовая выгрузка ID из Б24

**Через админку (основной способ):** **Настройки сайта** → **«Битрикс24: REST каталог»** — задайте вебхук и инфоблоки, сохраните. Кнопка **«Открыть сопоставление»** (на той же странице) или в левом меню **«Сопоставление каталога с Б24 (товары)»** открывает страницу с **пробным прогоном** (без записи) и **записью** в базу. Логика та же, что у команды ниже.

Команда (сервер / CI, опционально):

```bash
cd backend
python manage.py sync_bitrix_catalog_ids
```

Проверка вебхука без синка: на странице настроек REST каталога **«Проверка вебхука»** или пункт меню **«Проверка вебхука Битрикс24 (каталог)»** — `profile` и пробная выгрузка. Реализация проверки: `api/services/bitrix24_admin_test.py`.

Резерв без админки: переменные `BITRIX24_*` в `.env` (если поля в админке пустые). Приоритет для команды синка: аргументы CLI → **админка** → **env**.

Сопоставление с сайтом:

- **Вариант:** поле «Внешний код в Б24 (XML_ID), вариант»; если пусто — в качестве ключа используется **артикул WB** (`wb_nm_id`), если он совпадает с XML_ID в каталоге Б24.
- **Товар:** поле «Внешний код в Б24 (XML_ID)»; если пусто — **слаг** товара (если в Б24 внешний код задан так же).

По умолчанию заполняются только пустые `bitrix_catalog_id`; в админке включите «Перезаписать уже заполненный» или в CLI `--force`. Пробный прогон: кнопка в админке или `--dry-run`. В админке в «Дополнительно» — отключение `product.list` / `offer.list`, как `--no-products` / `--no-offers`.

Реализация: `api/services/bitrix24_rest.py`, `api/services/bitrix24_catalog_sync.py` (`run_bitrix24_catalog_sync_job`).

## Как протестировать

1. **Юнит-тесты:** `cd backend && python -m pytest tests/test_astrum_crm.py tests/test_bitrix_product_map.py tests/test_bitrix24_catalog_sync.py tests/test_bitrix24_admin_test.py -q`
2. **Ручной e2e:** задать тестовый ключ и `assigned` на dev, оформить заказ с витрины, в админке открыть заказ — статус «В Битрикс24» и заполненный ID при успехе; в панели Astrum — новая заявка; в Б24 — контакт/сделка по правилам приложения.
3. При **401** проверьте ключ; при **422** — тело ответа в `bitrix_sync_error` и сверку с [документацией Astrum](https://app-5.astrum.agency/documentation).

## Ограничения текущей версии

- Отправляются только заказы **корзины** (`CartOrder`). Заявки калькулятора и обратного звонка в Astrum **не** дублируются (при необходимости — отдельная задача).
- Фоновая очередь Celery не используется: запрос синхронный при создании заказа; при таймауте клиент всё равно получает 201, повтор — команда `retry_astrum_crm_orders`.
- **Стадия воронки в Б24 не отображается на сайте:** после создания заказа менеджер может двигать сделку в Битрикс24, но Django об этом не узнаёт — в админке остаются только локальные `fulfillment_status` / `payment_status`, пока их не меняют вручную на сайте.

## Технический долг: стадия сделки из Б24

**Цель:** показывать в админке (и при необходимости в ЛК) актуальную стадию сделки из Битрикс24.

**Варианты реализации (на выбор или в комбинации):**

1. **Исходящий вебхук Б24** на endpoint сайта (событие смены стадии сделки/лида) → обновление `CartOrder` по связке `bitrix_entity_id` (нужно уточнить, что именно возвращает Astrum в `id` — id заявки Astrum vs `DEAL_ID` в Б24, и есть ли стабильная связь с заказом на сайте).
2. **Периодический опрос REST** (`crm.deal.get` / аналог по типу сущности) по сохранённому ID сделки, если в БД хранится именно CRM-ID; либо поиск сделки по полю с номером заказа (`order_ref`), если ID сделки в ответе Astrum другой.
3. **Модель данных:** либо маппинг стадий Б24 → существующий `fulfillment_status`, либо отдельные поля вроде `bitrix_stage_id` / `bitrix_stage_name` + опционально `bitrix_stage_synced_at`, чтобы не ломать семантику локальных статусов.

**Зависимости:** секрет для проверки вебхука или вебхук с правами CRM; таблица/конфиг маппинга стадий; решение по источнику истины (только отображение vs запрет ручного конфликта со статусом на сайте).

Зафиксировано в бэклоге: [next-steps.md](./next-steps.md) → «Технический долг».
