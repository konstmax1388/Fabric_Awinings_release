## Заказы из корзины (`POST /api/leads/cart/`, модель `CartOrder`)

| # | Риск | Суть | Направление исправления |
|---|------|------|-------------------------|
| TD-ORDER-1 | Сумма не сверяется с составом | Поле `totalApprox` принимается с клиента без проверки против \(\sum (\text{priceFrom} \times \text{qty})\) по строкам. Возможны ошибка UI и намеренное занижение/завышение. | Валидация в `CartOrderCreateSerializer` (строгое равенство или допуск в рублях); при расхождении — 400. |
| TD-ORDER-2 | Нет проверки товаров и цен | Состав заказа не сопоставляется с БД: нет гарантии, что `productId`/`slug` существуют и что `priceFrom` совпадает с актуальной ценой витрины. | Опционально для «заявочного» режима; для магазина — разбор строк по `Product`/`Variant`, подстановка цены с сервера или отказ в оформлении при расхождении. |
| TD-ORDER-3 | Ошибка письма не видна клиенту | После создания заказа вызывается `notify_cart_order`; при сбое SMTP исключение глушится, ответ всё равно **201**. Клиент считает, что уведомление ушло менеджерам. | Логирование + метка на заказе (`notification_error`) или фоновая очередь с ретраями; при критичности — уведомление пользователю «заказ принят, если что-то срочное — позвоните». |
| TD-ORDER-4 | ~~Нет rate limit на оформление~~ **частично снято** | Ранее не было throttle. **Сделано:** `LeadSubmissionThrottle` (40/ч с IP) на `POST /api/leads/calculator/`, `callback/`, `cart/`; отдельный лимит на регистрацию. Риск массовых заказов с разных IP / обход лимита остаётся. | При необходимости — ужесточить лимиты, капча, идемпотентность (см. TD-ORDER-5). |
| TD-ORDER-5 | Двойная отправка заказа | Повторный сабмит формы или двойной клик создаёт несколько `CartOrder` с разными `order_ref`. | На фронте: блокировка кнопки до ответа (частично есть), debounce; на бэке: заголовок идемпотентности (`Idempotency-Key`) или одноразовый токен с TTL. |

---

*Документ можно дополнять новыми идентификаторами `TD-…` по мере аудита других подсистем.*
