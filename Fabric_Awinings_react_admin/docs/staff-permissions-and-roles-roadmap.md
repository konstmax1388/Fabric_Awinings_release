# Staff: роли на бэкенде и доступ к Django Admin — дорожная карта

**Статус:** зафиксировано для возврата к задаче; **реализация отложена** до выноса сайта на продакшен-хостинг (или отдельного эпика по безопасности).

**Дата фиксации:** 2026-04-14.

## Зачем этот документ

Нужно разделить возможности **менеджеров** (каталог, заказы, контент в React на `/staff/`) и **технических специалистов** (полный доступ к моделям и рискованным операциям в Django на `/admin/`). В интерфейсе React часть разделов уже скрыта, но **без проверок на API любой валидный staff JWT по-прежнему может вызывать все эндпоинты Staff API**, если знает URL.

## Текущее состояние в коде (зафиксировано)

- **Staff API** (`/api/staff/v1/…`): для ViewSet и APIView используется связка `IsAuthenticated` + **`IsStaffUser`** (`backend/api/permissions.py`).
- **`IsStaffUser`** проверяет только **`user.is_staff`** — отдельных Django-групп или permission-кодов на уровне разделов **нет**.
- Выдача JWT: `StaffTokenObtainPairView` — пользователь с `is_staff=True` может получить токен; дальнейшая гранулярность не применяется.
- **Django Admin**: доступ обычно у всех `is_staff=True` с соответствующими правами на модели; отдельной политики «только React, без Django» в этом репозитории **не описано в коде** (см. `DJANGO_ADMIN_ENABLED` и редиректы в [phase-8-django-admin-cutover.md](./phase-8-django-admin-cutover.md)).

## Целевое поведение (когда вернётесь к задаче)

1. **Группы или permission-коды** в Django, например (имена условные):
   - роль «операционная»: каталог + заказы + лиды + контент, **без** настроек сайта, **без** пользователей/групп, **без** service-actions (SMTP, Битрикс, WB и т.д.), либо с узким списком;
   - роль «техническая»: полный Staff API + вход в **Django Admin** (или отдельный флаг «разрешён вход в `/admin/`»).

2. **Staff API:** на каждый чувствительный ViewSet / `path` — свой permission-класс или проверка группы / `django.contrib.auth.models.Permission`, плюс базовый `IsAuthenticated`. Варианты реализации:
   - наследники `BasePermission` (`HasStaffGroup`, `DjangoModelPermissions` с кастомным маппингом);
   - или `django-guardian` / только группы — по согласованию с командой.

3. **Django Admin:**
   - либо **кастомный `AdminSite.has_permission`** / middleware: пользователи без группы «Django admin» получают 403 на `/admin/`;
   - либо **разделение учёток**: технические — `is_staff` + Django; менеджеры — отдельное поле вроде `can_use_staff_api` **без** права заходить в Unfold (потребует изменения условия выдачи JWT и логики `IsStaffUser`).

4. **Прод:** те же правила; при необходимости **ограничение по IP** для `/admin/` на reverse proxy.

5. **Тесты:** для каждой критичной группы — запросы под JWT пользователя с узкими правами → ожидаемые **403** на запрещённые маршруты.
